Types are the most basic building block of every program. We can say that underlying all computation that takes place around us everyday is made possible because of types. But what do we mean when we talk about types? For us this means focusing on a subset of the broader idea of types that exists in the world of programming known as primitive data types. And because we are learning about types in the context of the Apex programming language, we will start this exploration of types by looking at the primitive data types found in Apex.

What do we mean by primitive data type? Primitive tells us it is the most simple, data means information, and type tells us something descriptive about the nature of that data. Is the data a number? Or is it some kind of text? Or is it the idea of true or false? To make this abstract idea more concrete, let's look at a couple of examples.

Integers

Given the following expression:

{% highlight ruby %} 1 + 2 = 3 {% endhighlight %}

How might we describe this expression in terms of primitive data types?

We see there are three numbers: 1, 2 and 3. What type of numbers are they? In the primitive data types given to us by Apex, we would say numbers like 1, 2 and 3 are Integers.

Integers are numbers without a decimal point. As a primitive data type in various programming languages, Integers are usually expressed in terms of a range of a maximum low and a maximum high number. In Apex, we can represent Integers from -2,147,483,648 as the lowest possible number to 2,147,483,647 as the highest possible number. It is always preferable to use an Integer primitive data type when possible.

Strings

Let's look at another example, this time involving a different primitive data type:

{% highlight ruby %} 'There are single quotes around me' {% endhighlight %}

In this example, we don't see an expression but rather a statement. In this case, we would say the primitive data type of this statement is a String. Technically, a String is a sequence of Characters, but Apex does not support a Character primitive data type independent of String.

Strings are any characters surrounded by single quotes. There are a number of String operations we can perform such as converting a String of characters to all uppercase or lowercase characters, obtaining a substring of characters from a larger String, or even reversing a String. It's always good to consider how big any data is in terms of its memory footprint, but for now, let's keep things simple and say there is no limit to the size of a String we can define in Apex.

Expressions vs Statements

So far we've introduced Integers and Strings and the terms expression and statement. We said that this expression:

{% highlight ruby %} 1 + 2 = 3 {% endhighlight %}

is different than this statement:

{% highlight ruby %} 'There are single quotes around me' {% endhighlight %}

What do you think is the difference between an expression and a statement?

Before continuing lets clearly define expressions and statements, as they are important words we will use repeatedly going forward.

Expressions are anything that calculates a value or performs an operation. In very simple terms, expressions do things. The body of an expression tells us how the expression does what it does. We will explore this idea in much more detail when we talk about functions and methods as we move towards understanding Classes and Objects. But for now, the simple idea that expressions do things is perfectly fine.

Statements, unlike expressions, do nothing. Instead, Statements define facts or values. Statements can be used as inputs or in the body of expressions, but should not be thought of as the same thing as expressions. If you're ever unsure about whether or not something is a statement or an expression, ask yourself if some action is happening. If the answer is no, then you know it is a statement. The idea of statements in programming will have more significance when we talk about variables and assignment.

Booleans

Next in our introduction to primitive data types, let's look at the Boolean type. The term Boolean comes from George Boole and a system of mathematics credited to him known as Boolean Algebra. In Boolean Algebra, all variables are represented as one of two values:

{% highlight ruby %} true false {% endhighlight %}

Booleans refer two only two possible values, true or false. We use Booleans most commonly in defining control flow through programs in what is commonly referred to as conditional branching. For now though, you only need to remember that the Boolean primitive data type refers to only two values.

Booleans vs Strings

In the previous sections we defined the String primitive data type as any group of characters surrounded by single quotes, and Boolean primitive data types as being one of two values: true or false. But how would we describe the following statements in terms of primitive data types?

{% highlight ruby %} 'true' 'false' {% endhighlight %}

In this case, both of these statements are Strings. We can say they are String representations of Boolean values, but their primitive data type is still just a String. The rule that Strings are any characters wrapped with single quotes here demonstrates the sometimes subtle distinctions between types, as we will see as we dive deeper into the remaining number primitive data types.

Decimals

The Decimal primitive data type is the type most commonly used to represent money values.* The following examples are expressions using the Decimal primitive data type:

{% highlight ruby %} 5.00 + 7.50 = 12.50 1.0000000000001 + 0.0000000000000 = 1.0000000000001 {% endhighlight %}

Decimals are numbers that contain a decimal point. In the context of Apex, Decimal numbers are said to be of arbitrary precision meaning there is not a pre-defined limit to how many decimal places the number can have.

*Depending on the context, decimals may not be the best way to calculate or store money values. Many people agree that dealing with money is best done with Integers to avoid problems with rounding and the strange behavior sometimes witnessed when needing to truncate decimals.

Doubles

Like Decimals, Doubles also have decimal points. The distinction between a Double and a Decimal is a lower-level idea concerning the amount of memory that is allocated when a Decimal vs a Double primitive data type is used. For now, we are not going to worry about Doubles, but you can think of Doubles as extra big Decimals for now.

Longs

Like Doubles, we are not going to worry about Longs at this time. In the same way Doubles are related to Decimals, Longs are related to Integers. For now, think of Doubles as extra big Integers that consume more memory. We won't need to use Longs or Doubles for some time, if ever. What is important at this time is to recognize that Doubles are extra big Decimals, and Longs are extra big Integers.

The Scourge Of Programming: Time

Programming applications for the real world inevitably deals with some notion of time, dates, or the combination of time and dates, datetimes. In most programming languages concepts of Date or Time or DateTime are regarded as primitive data types and Apex is no exception. Apex provides us with the primitive data types of Date, Time and DateTime.

Date

The Date primitive data type represents a date without the notion of time. Let's take a look at a couple of examples of Date statements:

{% highlight ruby %} 2015/06/18 06/18/2015 2015-07-08 12-12-12 {% endhighlight %}

Dates only represent the notion of a date independent of time, and can be written as statements in a variety of ways. We will explore ways to format dates when we talk about Date objects. We will place special emphasis on removing any ambiguity in our Date statements, e.g. 12-12-12, by remembering to always format dates in a way that removes the possibility of misinterpretation of the values representing months, days and years.

Time

Time is as you would expect a primitive data type that allows us to express hours, minutes, seconds and even finer levels of granularity (depending on the language and the context, we can express time to an arbitrary precision similar to Decimals). In the context of Apex we can use the Time primitive data type to represent hours, minutes, seconds and milliseconds as the lowest level of granularity. Here are a variety of Time statements:

{% highlight ruby %} 05:15:00 15:20:20 10:10:10:10 {% endhighlight %}

Times let us represent data in terms of hours, minutes, seconds and milliseconds within Apex. Time can be thought of in terms of a 24 hour clock, or a 12 hour clock using the AM and PM designation. As programmers it is highly recommended that you become comfortable with the 24 hour clock because it is frequently the default mode for representing time.

DateTime

As you've probably already guessed the DateTime primitive data type combines Dates and Time together as one type. However, there is an additional piece of information that needs to be considered when dealing with DateTime values, and it is the scourge of programming known as time zones.

We will talk about programming with time zones when the time is right, but for now all you need to focus on is the DateTime primitive data type and the way it is represented as a statement or value. Let's take a look a few examples of how DateTime values can be represented:

{% highlight ruby %} 2015-06-18 05:32:24 2015-06-17 00:00:00 22:37:51.703Z 1434605952372 {% endhighlight %}

DateTimes conceptually are simple and easy to grasp, but is likely the most difficult primitive data type to work with. DateTimes are complicated by the fact that they can be represented in terms of a date and time: 2015-06-18 05:32:24, or they can be represented in terms of their Date component only: 2015-06-17 00:00:00. Or in some contexts we may only care about the Time component: 22:37:51.703Z. The 'Z' at the end of the Time indicates the Zulu time zone which corresponds with the Greenwich Mean Time (GMT). Although there are other possible representations of DateTime values, the last example above is the number of milliseconds since January 1, 1970 00:00:00Z. Why is that important? January 1, 1970 is called Unix time or epoch time and provides a universally consistent representation of time regardless of time zone, operating system or platform. We will not likely use the epoch time in these sessions, but it serves to illustrate a little about the complexity that surrounds programming with dates and time.

Wrapping Up

Now that you're familiar with the basic primitive data types offered in Apex, we can generalize what we've learned about primitive data types in terms of four broad categories:

  1. Numbers
  2. Text
  3. Time
  4. Booleans

Although we learned there are multiple Number primitive data types, the two we will focus on are the Integer and Decimal primitive data types. We learned that Strings are any characters surrounded by single quotes (it's important in the context of Apex to not use double-quotes), and we also learned that entire systems of mathematics can be created by describing expressions merely in terms of true and false with the Boolean primitive data type. Lastly, we looked at Dates, Times and their combined primitive data type, DateTime, and also looked at their various representations and peeled back a few layers of the complexities surrounding programming with Dates and Times.

This brief introduction to primitive data types has hopefully provided you with a model for understanding the basic building blocks of programs and prepares you to begin exploring how to construct these primitive data types in Apex. The goal is to give you the mental model needed to be able to correctly identify primitive data types and also identify what primitive data type to use in different situations. The challenge questions below will provide you with some application practice so you can verify your understanding.

If you would like to read more about the data types in Apex, a detailed, complete list of types offered by Apex can be viewed here. We did not cover all of the primitive data types offered by Apex just yet, because not all of Apex's primitive data types are necessary for understanding how to program. But the majority of Apex's primitive data types are found in just about every programming language, and the primitive data types we've covered in this session are immediately transferable to most languages.
